## 1. Purpose

The PMO Dashboard is the central operational system for managing memo requests, approval workflow, project spending, software license tracking, hardware procurement tracking, and PMO master data.

The system is designed to support PMO operations first. Authentication, permission enforcement, and notification routing will be implemented later by the Tech Team, but the business rules must already define who should be able to do what.

---

## 2. Core Design Principles

1. Memo is the source of truth.
2. Only Approved memos create downstream business impact.
3. Approved memos are immutable.
4. Business records should remain auditable even when removed from the UI.
5. No submitted business record should be physically deleted from the database.
6. Downstream modules must consume approved memo data only.
7. PMO override is allowed only with reason and evidence.
8. Business logic takes priority over UI convenience.
9. Master data should be reusable, but selected fields may support controlled free text.
10. Every important transition must be traceable through audit logs.

## 3. Core Modules

### 3.1 Create Memo

Used to create memo requests.

Memo can be saved as Draft or submitted into the approval flow.

Supported memo types:

- License
- Hardware
- Internal team activity
- Customer entertainment
- Deployment expense

Memo number is not generated by the system. It is entered manually because the memo number may come from an external request process.

System rule:

- Memo Number must be unique.
- Duplicate memo number is not allowed.
- When duplicating a memo, the Memo Number must not be copied.
- User must enter a new Memo Number.

---

### 3.2 Pending Memo

Used to manage submitted memos that are waiting for approval.

Pending memo may be:

- Approved
- Rejected
- Cancelled by requester or PMO
- Overridden by PMO

Pending memo PDF can be downloaded because it may be used for external approval such as email approval.

---

### 3.3 All Memo

Used to view memos across all statuses.

Statuses include:

- Draft
- Pending
- Approved
- Rejected
- Cancelled
- Voided

All Memo is also where approved memo can be viewed, duplicated, downloaded, and linked to budget pool tagging when applicable.

---

### 3.4 Budget & Spend

Used to track financial impact.

Only Approved memos are counted as Actual Spend.

Voided memos must not be counted in Actual Spend calculations.

Supported currency:

- THB only, for now.

Decision (2026-07-03): USD support was reverted. There is no confirmed USD use case in current
PMO workflow, and multi-currency created unnecessary complexity across Budget & Spend, Actual
Spend, PDF, Export, Forecast, and downstream modules. Currency is still stored explicitly at
record level (a dormant field, always THB) so USD can be reintroduced later without a schema
change if a confirmed use case emerges.

---

### 3.5 License Management

Used mainly for License memos.

An approved License memo may contain multiple software line items.

License Management should display license data by software line item, not only by memo header.

Main areas:

- License index / expiry tracking
- License summary by software, project, and seat
- User license mapping
- PMO review queue for account list from approved memos
- Other subscription tracking

Other subscription is informational only and does not flow into Budget & Spend.

---

### 3.6 Device Management

Used mainly for Hardware memos.

Approved Hardware memo creates Purchase Order records.

Purchase Order tracks purchase progress until devices arrive.

When devices arrive, Device Registry records are created per physical device.

Manual device add and bulk upload are allowed and do not require memo number.

---

### 3.7 Settings

Used by PMO to manage master data and configurable lists.

Confirmed master data:

- Project / Team
- Approver / Reviewer
- Device Type
- Software master list, with PMO approval before new free-text software becomes master data

Budget Pool settings may remain inside Budget & Spend because only PMO should access that function.

---

### 3.8 Resource Management

Resource Management is out of the current implementation scope and will be merged later from a separate implementation.

## 4. High-Level Data Flow

### 4.1 Memo Flow

```text
Create Memo
  ├─ Save Draft
  └─ Submit
       ↓
     Pending
       ↓
Approved / Rejected / Cancelled
```

Only Approved memos create downstream data.

---

### 4.2 License Memo Flow

```text
Approved License Memo
  ↓
License Line Items
  ↓
License Index / Expiry Tracking
  ↓
License Summary
  ↓
User License Mapping
  ↓
Budget & Spend
```

If account list is provided in the memo, it enters PMO review before being added to User License Mapping.

---

### 4.3 Hardware Memo Flow

```text
Approved Hardware Memo
  ↓
Purchase Order
  ↓
Purchase Tracking
  ↓
Partially Arrived / Arrived
  ↓
Device Registry
  ↓
Budget & Spend
```

---

### 4.4 Other Memo Types

```text
Approved Internal / Entertainment / Deployment Memo
  ↓
Budget & Spend
```

## 5. Standard Metadata

All major entities should store standard metadata in the backend:

- Created By
- Created Date
- Updated By
- Updated Date

This applies to:

- Memo
- Budget Pool
- Manual Spend
- License records
- Device records
- Purchase Order records
- Master Data records
- User License Mapping
- Other Subscription records

These fields do not need to be shown in every UI screen, but they must be available for audit and troubleshooting.

---

## 6. Delete Policy

Only Draft memo can be deleted from the user perspective.

However, delete should be implemented as soft delete.

Deleted records should be removed from normal UI views but retained in the database for audit.

Recommended backend fields:

- Deleted flag
- Deleted By
- Deleted Date
- Delete Reason, if applicable

---

## 7. Time Zone

System business timestamp should use Bangkok time.

Display timezone:

- Asia/Bangkok

Implementation may still use a consistent backend storage standard, but business-facing date and time must be interpreted and displayed in Bangkok time.

---

## 8. Audit and Versioning

The system does not need formal version history.

Changes should be captured through audit logs instead.

Audit logs should be used for:

- Submit
- Approve
- Reject
- Cancel
- Override
- Void
- Delete draft
- Duplicate
- Budget tagging
- License user assignment
- Device status changes
- Master data changes

---

## 9. Out of Scope for Current Functional Build

The following are planned for Tech Team or later implementation:

- Authentication
- Role-based permission enforcement
- Notification routing
- Email approval automation
- Login
- Full user identity model

However, current business rules must still define the intended actor for each action.
